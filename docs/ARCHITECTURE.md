## 1. System Architecture Overview

### 1.1 Architecture Pattern: Modular Monolith

LedgerPulse follows the **Modular Monolith** pattern — a single deployable unit with logically separated modules (services) that communicate through in-process method calls. This is deliberately chosen over microservices for Phase 0–4 to maximize development velocity while maintaining clean domain boundaries. The modules are structured such that extraction into standalone microservices is possible if scale demands it.

## 3. Service Decomposition

Every service below is a **directory within `packages/backend/src/services/`** — a collection of related functions and classes sharing a common concern.

### 3.1 Document Ingestion Service

**Purpose:** Accept documents via multiple channels, validate file types, store original files, enqueue OCR jobs.

| Method | Channel | Implementation |
|--------|---------|---------------|
| `POST /api/documents/upload` | Web upload | Express + multer |
| `EmailIngestionJob` | Email (IMAP) | `node-imap` / `mailparser` polling cron job |

**Responsibilities:**
- File type validation (PDF, JPG, PNG, TIFF — max 10MB)
- Virus scanning placeholder (ClamAV integration point)
- Store raw file to local filesystem or S3-compatible storage
- Create `Invoice`, `DeliveryNote`, or `EWayBill` record with `PENDING` status
- Enqueue OCR job to Bull queue

**Files:** `src/services/ingestion/`, `src/jobs/ocr.job.ts`

### 3.2 OCR & Extraction Service

**Purpose:** Convert document images/PDFs to structured data — line items, totals, dates, tax amounts, vendor info.

**Approach (ADR-003):** Local Tesseract.js for Phase 1 with a cloud OCR fallback (Google Document AI / Azure Form Recognizer) as an optional premium tier.

**Pipeline:**
```
Raw File → Preprocessing → OCR → Post-processing → Entity Extraction → Validation
             │                │          │                │                │
         OpenCV.js      Tesseract.js  Regex/NLP     Zod schema      Confidence
         (deskew,       (text         (clean        validation      threshold
          denoise,       extraction)   whitespace,                     check
          threshold)                    normalize)
```

**Files:** `src/services/ocr/`, `src/jobs/extraction.job.ts`

### 3.3 3-Way Matching Engine

**Purpose:** The core differentiator. Compares three data sources line-by-line and produces a match score with itemized discrepancies.

**Match Algorithm (simplified):**

```
                    ┌─────────────────────────────┐
                    │  Step 0: Group by Vendor +   │
                    │  approximate date range      │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────▼───────────────┐
                    │  Step 1: Invoice ↔ Delivery  │
                    │  Note (2-way)                │
                    │                              │
                    │  - Total amount match (±1%) │
                    │  - Line item quantity match  │
                    │  - Unit price match          │
                    │  - Weight match (if DN has   │
                    │    weighbridge value)        │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────▼───────────────┐
                    │  Step 2: Invoice+DN ↔ E-Way  │
                    │  Bill (3-way)                │
                    │                              │
                    │  - Total value match (±2%)  │
                    │  - GSTIN match               │
                    │  - Tax amount match          │
                    │  - HSN/SAC code consistency  │
                    │  - Transport mode match      │
                    └─────────────┬───────────────┘
                                  │
                    ┌─────────────▼───────────────┐
                    │  Step 3: Score Calculation   │
                    │                              │
                    │  matchScore = weighted avg   │
                    │  of all comparison fields    │
                    │                              │
                    │  > 0.95 → MATCHED            │
                    │  0.70–0.95 → PARTIAL         │
                    │  < 0.70 → MISMATCH           │
                    └─────────────┬───────────────┘
                                  │
                                  ▼
                         Write MatchResult record
                         Store discrepancies JSON
                         Trigger notification if MISMATCH
```

**Configurable thresholds** stored in `MatchResult.matchScore` — adjustable per tenant in multi-tenant Phase 5.

**Files:** `src/services/matching/`

### 3.4 Tax Log Integration Service

**Purpose:** Sync E-Way Bill data from government GST portal (India) or equivalent tax APIs.

**Responsibilities:**
- Schedule periodic polling (every 6 hours via `node-cron`)
- Authenticate with govt API (GST portal API credentials)
- Upsert E-Way Bill records by `ewayBillNumber`
- Link to existing invoices by GSTIN + value range matching
- Handle rate limiting and retry with exponential backoff (govt APIs are notoriously unreliable)

**Files:** `src/services/tax/`, `src/jobs/ewaybill-sync.job.ts`

### 3.5 Discrepancy & Approval Service

**Purpose:** Manage the lifecycle of flagged discrepancies — review, resolve, or escalate.

**States:**
```
PENDING → REVIEWED → RESOLVED
                  ↘ ESCALATED → CLOSED
```

**Responsibilities:**
- List discrepancies grouped by severity (MISMATCH > PARTIAL)
- Allow reviewer to add notes, adjust match results manually
- Generate dispute draft PDF (pre-populated reconciliation statement)
- Track audit trail (`reviewedBy`, `reviewedAt`, notes)

**Files:** `src/services/discrepancy/`, `src/services/approval/`

### 3.6 Notification Service

**Purpose:** Alert stakeholders when discrepancies are found or approvals are pending.

**Channels (Phase 1 → Phase 5):**
| Channel | Phase | Implementation |
|---------|-------|---------------|
| In-app notification | 1 | WebSocket or polling from dashboard |
| Email | 2 | Nodemailer + SMTP (SendGrid/Mailgun) |
| WhatsApp (Future) | 5 | Twilio API |

**Files:** `src/services/notification/`

## 4. Data Flow Design

### 4.1 Invoice Ingestion Flow (Email → OCR → Extraction → Storage)

```
User sends email with invoice PDF attachment
         │
         ▼
Email Polling Job (runs every 5 min)
  - Connects via IMAP
  - Filters unseen emails from known vendor domains
  - Downloads attachments
  - Validates file type & size
         │
         ▼
Ingestion Service
  - Generates UUID filename (avoids collisions)
  - Saves file to `uploads/invoices/{vendorId}/{uuid}.pdf`
  - Creates `Invoice` record with status=PENDING, filePath=...
  - Enqueues OCR job with { invoiceId, filePath }
         │
         ▼
Bull Queue → OCR Job Processor
  - Loads file from disk
  - Runs Tesseract.js OCR
  - Stores rawText on Invoice record
         │
         ▼
Extraction Job Processor (chained after OCR success)
  - Parses rawText with regex patterns:
    - Invoice number pattern: /INV[-]\d+/i
    - Total amount: /Total[\s:]+₹?(\d+[.,]\d{2})/
    - Date: /\d{2}[-/]\d{2}[-/]\d{4}/
    - Line items: per-line parsing with quantity × rate
  - Runs Zod schema validation
  - Updates Invoice.status = PROCESSED
  - Updates Invoice.totalAmount, taxAmount, lineItems (JSON)
  - Checks confidence score; if < 0.8, flags for manual review
         │
         ▼
If confidence ≥ 0.8:
  - Triggers 2-way matching attempt (if delivery note exists for same vendor+date)
  - Notification: "Invoice {number} processed — ready for matching"
```

### 4.2 3-Way Matching Flow

```
Trigger: New Invoice PROCESSED or Delivery Note PROCESSED or E-Way Bill synced
         │
         ▼
Matching Engine
  Step 0 — Group candidates:
    SELECT invoices, deliveryNotes, ewayBills
    WHERE vendorId = X
      AND ABS(julianday(invoiceDate) - julianday(deliveryDate)) <= 7
      AND status IN ('PROCESSED')
    ORDER BY date DESC
         │
         ▼
  Step 1 — 2-way match (Invoice ↔ Delivery Note):
    - Parse Invoice.lineItems JSON
    - Parse DeliveryNote.lineItems JSON
    - Compare line-by-line by description/HSN code
    - For each matching line:
      ✓ Quantity match? (±5% tolerance)
      ✓ Unit price match? (±2% tolerance)
      ✓ Total = qty × rate?
    - Aggregate totals: |invoice.total - dn.total| / max(total)
    - Weighbridge check: if deliveryNote.weightbridgeValue exists,
      compare with invoice line-item weights
         │
         ▼
  Step 2 — 3-way match (+ E-Way Bill):
    - Fetch linked EWayBill records
    - Compare:
      ✓ EWayBill.totalValue ≈ Invoice.totalAmount (±2%)
      ✓ EWayBill.fromGstin ≈ Vendor.gstin
      ✓ Tax amounts align (CGST + SGST = total tax)
    - HSN/SAC code consistency (if available in all three)
         │
         ▼
  Step 3 — Score & persist:
    matchScore = weighted average(
      0.35 × totalAmountMatch,
      0.30 × lineItemMatch,
      0.20 × taxMatch,
      0.15 × weightMatch (if applicable)
    )
    
    INSERT MatchResult {
      invoiceId, deliveryNoteId, ewayBillId,
      matchScore,
      status: score > 0.95 ? 'MATCHED' : score > 0.70 ? 'PARTIAL' : 'MISMATCH',
      discrepancies: [ { field, expected, actual, severity } ]
    }
         │
         ▼
  Post-match:
    - If MATCHED: update Invoice.status = MATCHED, DeliveryNote.status = MATCHED
    - If PARTIAL/MISMATCH: update to DISCREPANCY status
    - Enqueue notification job
```

### 4.3 Discrepancy Resolution Flow

```
User logs into Dashboard
         │
         ▼
Discrepancy List (filterable by:
  - Status (PENDING, REVIEWED, RESOLVED)
  - Severity (PARTIAL, MISMATCH)
  - Vendor, date range)
         │
         ▼
User clicks a discrepancy → Detail View:
  - Side-by-side comparison: Invoice vs Delivery Note vs E-Way Bill
  - Highlighted mismatched fields in red
  - Match score, individual field scores
  - Original document preview (PDF/image viewer)
         │
         ▼
User actions:
  ┌────────────────────────────────────────────────────┐
  │  [Mark Resolved] - Accept with override note       │
  │  [Request Re-upload] - Triggers notification to    │
  │    vendor for corrected document                   │
  │  [Generate Dispute] - Creates pre-populated        │
  │    dispute draft PDF with mismatch details         │
  │  [Escalate] - Flags for senior reviewer            │
  └────────────────────────────────────────────────────┘
         │
         ▼
On Resolve:
  - MatchResult.reviewedBy = user.id
  - MatchResult.status = 'RESOLVED'
  - Update Invoice.status = RESOLVED
  - Audit log entry created
  - If dispute generated: save PDF path, log
```

### 4.4 E-Way Bill Sync Flow

```
Cron Job (runs every 6 hours)
         │
         ▼
Tax Log Integration Service
  - Read last sync timestamp from SyncLog table
  - Authenticate with GST portal API
    (OAuth2 client_credentials grant)
         │
         ▼
API Request:
  GET /api/ewaybill?fromDate={lastSync}&toDate={now}
  Headers: Authorization: Bearer {access_token}
         │
         ▼
Response processing:
  - For each EWB in response:
    - Check if ewayBillNumber exists → UPSERT
    - Parse rawData (store full JSON)
    - Set status = PROCESSED
  - On rate limit (429):
    - Exponential backoff: wait 2^n seconds, retry up to 3 times
  - On auth failure:
    - Alert admin (notification service)
    - Log error, skip cycle
         │
         ▼
Post-sync:
  - Update SyncLog with last successful timestamp
  - Trigger 3-way matching for any invoices linked to new/updated EWBs
```

## 5. Database Architecture

### 5.1 Schema Design Decisions

**Current schema** (Phase 0) — `packages/backend/prisma/schema.prisma`:

```
Vendor ──1:N── Invoice ──1:N── MatchResult
  │               │
  │               └──1:N── DeliveryNote
  │
  └──1:N── DeliveryNote
```

**Key design decisions:**

| Decision | Rationale |
|----------|-----------|
| **UUID primary keys** | Avoids sequential ID leaks; safe for multi-tenant sharding (Phase 5) |
| **JSON strings for lineItems** | Line item structures vary per vendor; JSON schema is validated via Zod at the application layer rather than enforcing rigid relational columns |
| **Unique constraints on (vendorId, invoiceNumber)** | Prevents duplicate invoice ingestion for the same vendor |
| **Separate DocumentStatus enum** | Shared across Invoice, DeliveryNote, EWayBill — ensures consistent state machine |
| **Nullable foreign keys on MatchResult** | A match can be 2-way (Invoice+DN) or 3-way (+EWB); not all documents may exist at match time |
| **matchScore as Float (0.0–1.0)** | Allows configurable thresholds without schema changes; thresholds live in app config |
| **rawText for OCR output** | Preserves original extraction for re-processing without re-running OCR |
| **discrepancies as JSON string** | Flexible discrepancy detail structure per vendor rules; validated at application layer |

### 5.2 Indexing Strategy

```
Invoice:
  @@unique([vendorId, invoiceNumber])   ← Lookup by vendor + invoice
  @@index([status])                      ← Filter by status (dashboard)

DeliveryNote:
  @@unique([vendorId, deliveryNoteNumber])
  @@index([status])

EWayBill:
  @@unique([ewayBillNumber])            ← Dedup on govt ID
  @@index([status])

MatchResult:
  @@index([status])                      ← Filter by match outcome
  @@index([invoiceId])                   ← Lookup by parent invoice
  @@index([createdAt])                   ← Time-based dashboard queries
```

**Future indexes (Phase 2+):**
- `Invoice(invoiceDate)` — date-range filtering on dashboard
- `DeliveryNote(deliveryDate)` — same
- `MatchResult(invoiceId, status)` — composite for status-by-invoice queries

### 5.3 Migration Approach: SQLite Dev → PostgreSQL Prod

**Strategy: "Dev/Test on SQLite, Prod on PostgreSQL"**

This works because Prisma abstracts the database dialect at the query level, but there are **known differences** to manage:

| Dimension | SQLite (Dev) | PostgreSQL (Prod) | Mitigation |
|-----------|-------------|-------------------|------------|
| **Enum** | Simulated as strings | Native enum types | Prisma handles this automatically |
| **DateTime** | No timezone info | TIMESTAMPTZ | Use `DateTime` with UTC everywhere |
| **Floats** | REAL (4-byte) | DOUBLE PRECISION | Use `Float` in Prisma → maps to REAL in SQLite, DOUBLE in PG |
| **Migrations** | `prisma migrate dev` | `prisma migrate deploy` | Use the same migration files |
| **Case sensitivity** | Case-insensitive by default | Case-sensitive | Use `collate` or `citext` extension in PG |
| **Full-text search** | FTS5 extension | `tsvector` | Abstract search behind service layer |

**CI/CD Pipeline:**
```
Dev machine:  SQLite (DATABASE_URL=file:./dev.db)
CI pipeline:  SQLite (fast, no external service needed)
Staging:      PostgreSQL (same URL format, different provider in schema)
Production:   PostgreSQL (RDS / Railway / Fly.io managed PG)
```

**`prisma/schema.prisma` switching pattern:**

```prisma
// Default: SQLite for local dev
datasource db {
  provider = "sqlite"
  url      = env("DATABASE_URL")
}

// Production override in CI/CD:
// Set DATABASE_URL=postgresql://...
// AND override datasource provider via:
//   PRISMA_DATASOURCE_PROVIDER=postgresql
//   prisma generate --schema=prisma/schema.prisma --data-proxy
```

Alternatively, maintain separate schema files or use `prisma migrate` with `--schema` flag pointing to a PostgreSQL-specific override. For simplicity in Phase 0–2, we recommend a **single schema with manual provider switching** at deployment time:

```bash
# Dev
DATABASE_URL="file:./dev.db" npx prisma migrate dev

# Prod
sed -i 's/provider = "sqlite"/provider = "postgresql"/' prisma/schema.prisma
DATABASE_URL="postgresql://..." npx prisma migrate deploy
```

**Migration files** are stored in `prisma/migrations/` and are SQL-dialect specific. Separate migration directories for SQLite and PostgreSQL may be needed if complex migrations (e.g., extensions, triggers) diverge. For Phase 0–4, a single migration set targeting PostgreSQL is acceptable — SQLite dev can use `prisma db push` instead of `prisma migrate dev`.

## 6. API Design

### 6.1 Route Structure

```
Base URL: /api/v1

Health:
  GET    /health                    → Server + DB status

Documents (Ingestion):
  POST   /documents/upload          → Upload invoice/delivery note file
  POST   /documents/upload-email    → Trigger email re-ingestion (manual)
  GET    /documents/:id             → Get document metadata
  GET    /documents/:id/raw         → Get raw OCR text
  GET    /documents/:id/preview     → Serve original file (S3/local proxy)

Invoices:
  GET    /invoices                  → List (paginated, filterable by vendor/status/date)
  GET    /invoices/:id              → Single invoice detail
  PATCH  /invoices/:id              → Update invoice metadata (admin)

Delivery Notes:
```

| HTTP Code | Meaning | When |
|-----------|---------|------|
| `200` | Success | GET, PATCH, POST (sync) |
| `201` | Created | POST — resource created |
| `400` | Bad Request | Zod validation failure, malformed input |
| `401` | Unauthorized | Missing/invalid JWT |
| `403` | Forbidden | Insufficient role permissions |
| `404` | Not Found | Resource ID doesn't exist |
| `409` | Conflict | Duplicate invoice number for vendor |
| `413` | Payload Too Large | File exceeds 10MB |

**Error handling middleware pattern** (already in `src/index.ts`):
- Catch-all error handler at the app level
- Route handlers wrap async errors with `express-async-errors` or manual `next(err)`

### 6.4 Authentication Approach (Phase 5)

**Chosen: JWT with refresh tokens (see ADR-005)**

```
POST /auth/login
  Body: { email, password }
  Response: { accessToken (15min), refreshToken (7d) }

Middleware:
  Authorization: Bearer <accessToken>
  - Verify JWT signature (HS256 or RS256)
  - Extract user id + role from payload
  - Attach to req.user
  - Role-based guards: @requireRole('admin', 'reviewer')

Refresh:
  POST /auth/refresh
    Body: { refreshToken }
    Response: { accessToken, refreshToken (rotated) }
```

**Multi-tenant considerations (Phase 5+):**
- JWT contains `tenantId` claim
- All queries scoped by `WHERE tenantId = req.user.tenantId`
- Prisma middleware or row-level security via PostgreSQL (future)

## 7. Quality Attributes

### 7.1 Scalability

| Concern | Strategy |
|---------|----------|
| **Horizontal scaling** | Modular monolith can be replicated behind a reverse proxy (Caddy/NGINX); stateless Express app shares nothing except DB and Redis |
| **OCR processing** | CPU-intensive OCR runs in separate Bull queue workers; can scale by adding worker processes on separate machines consuming the same Redis queue |
| **Database** | SQLite is single-writer only → limits horizontal scaling in dev. PostgreSQL with connection pooling (PgBouncer) enables concurrent read replicas in prod |
| **File storage** | Start with local filesystem; migrate to S3-compatible (MinIO for self-host, AWS S3 / DigitalOcean Spaces for cloud) |
| **Rate limiting** | Express-rate-limit in-memory for single-instance; Redis-based store for distributed rate limiting across instances |

**Scaling tiers:**
```
Tier 1 (Phase 0–2): Single server, SQLite, local filesystem
Tier 2 (Phase 3–4): Single server + PostgreSQL + Redis (separate processes)
Tier 3 (Phase 5+):  Multiple API replicas + read replicas + S3 + workers
```

### 7.2 Reliability

| Concern | Strategy |
|---------|----------|
| **Job failure** | Bull queue auto-retries failed jobs (3 attempts with exponential backoff); dead-letter queue for permanently failed jobs |
| **OCR failure** | If confidence < 0.5 after retries, flag for manual OCR review via dashboard |
| **E-Way Bill API failure** | Exponential backoff (2, 4, 8 seconds); skip cycle after 3 failures; alert admin via notification service |
| **Database outage** | Prisma connection pool with retry; health endpoint probes DB; if DB is down, return 503 immediately rather than hanging |
| **Graceful shutdown** | Express `process.on('SIGTERM')` → close Prisma, disconnect Redis, finish in-flight requests (with timeout) |
| **Idempotency** | Upload endpoint uses UUID dedup; Bull jobs are idempotent (check status before processing) |

### 7.3 Security

| Concern | Implementation |
|---------|---------------|
| **Input validation** | Zod schemas on every request body; strict JSON parsing with `express.json({ limit: '10mb' })` |
| **File upload validation** | MIME type whitelist (`application/pdf`, `image/jpeg`, `image/png`); magic number check; max file size |
| **Rate limiting** | 100 requests per 15 minutes per IP (configurable); stricter for auth endpoints |
| **Helmet headers** | 11 security headers set by `helmet()` — CSP, HSTS, X-Frame-Options, etc. |
| **CORS** | Whitelist frontend origin(s); restrict HTTP methods |
| **SQL injection** | Prisma parameterized queries prevent injection by default |
| **JWT security** | Short-lived access tokens (15 min); refresh tokens stored in HTTP-only cookies or secure local storage |
| **Secrets management** | All secrets in `.env` — never committed; `NODE_ENV` controls error verbosity |
| **Dependency scanning** | `npm audit` in CI; Dependabot for automated PRs |

### 7.4 Performance

| Concern | Strategy |
|---------|----------|
| **Query optimization** | Prisma `select` to fetch only needed fields; `include` sparingly; pagination on all list endpoints |
| **Caching** | Redis for: vendor lookup cache (5 min TTL), session store, rate limit store |
| **OCR performance** | Queue-based processing means API doesn't block on OCR; Tesseract.js runs in worker thread |
| **Dashboard queries** | Materialized summary table or Prisma aggregations; pre-computed stats refreshed on matching events |
| **Lazy loading** | Large fields (rawText, lineItems JSON) fetched on detail views, not in list queries |
| **Connection pooling** | Prisma manages a configurable connection pool (default: 10) |

### 7.5 Observability

| Concern | Implementation |
|---------|---------------|
| **Structured logging** | Pino or Winston with JSON output; log level per environment (debug dev, info prod) |
| **Request tracing** | Unique `requestId` via middleware (UUID per request); included in error responses and logs |
| **Health endpoint** | `GET /api/health` returns server + DB status; consumed by load balancer / Docker health check |
| **Job monitoring** | Bull Board (Express middleware) for real-time queue visualization during development |
| **Metrics (Phase 5)** | Prometheus metrics endpoint (`/api/metrics`) — request count, latency percentiles, queue depth, DB pool size |
| **Error tracking** | Sentry integration point: capture unhandled exceptions with stack traces and request context |

## 8. Architecture Decision Records

### ADR-001: Modular Monolith vs Microservices

**Status:** Accepted

**Context:** LedgerPulse has clearly bounded domains (ingestion, OCR, matching, tax sync, approval). Microservices would allow independent scaling of the OCR pipeline (CPU-intensive) and independent deployability. However, the team is a solo developer (Ravi) and the product is pre-revenue.

**Decision:** Use a **Modular Monolith** in Phase 0–4. All modules live in `packages/backend/src/services/` as logically separated directories. Communication is via in-process function calls, not HTTP/RPC. Extraction to microservices is a documented future path.

**Trade-offs:**
| Pro | Con |
|-----|-----|
| Single deployable — fast iteration | Cannot scale OCR independently without extraction |
| No network overhead between services | Single failure domain (one crash kills all) |
| Simple local development (one `npm run dev`) | Technology lock-in (all in Node.js) |
| No API versioning between modules | Requires disciplined module boundaries to avoid spaghetti |

**Extraction trigger criteria:** When any of these are true:
- OCR worker CPU consistently > 80% during normal operation
- Number of Express route files exceeds 40
- Need to deploy a critical matching fix without redeploying OCR

### ADR-002: SQLite Dev → PostgreSQL Prod via Prisma

**Status:** Accepted

**Context:** The development environment should be zero-install (no Postgres daemon) and fast for CI. Production needs ACID compliance, concurrent writes, and managed backups.

**Decision:** Use **SQLite** for local development and test via `prisma db push` (schema drift allowed in dev). Use **PostgreSQL** in production via `prisma migrate deploy` (migration-controlled). Single Prisma schema with provider switching at deploy time.

**Trade-offs:**
| Pro | Con |
|-----|-----|
| Zero-config local dev — no Postgres install | SQLite ↔ PG type differences can cause subtle bugs |
| CI runs fast — no external DB service needed | Some Prisma features differ per provider (e.g., `@default(now())` semantics) |
| Prisma handles 95% of dialect differences | Need to test on PG before production deployment |
| Single schema file — no duplication | Migration files are SQL-dialect specific |

**Mitigations:**
- CI runs a "PG compatibility check" job using a Docker Postgres container on PRs
- Schema provider switching is scripted in `scripts/switch-db.sh`
- All date/time handled in UTC to avoid timezone drift

### ADR-003: OCR Strategy — Local Tesseract.js vs Cloud API

**Status:** Accepted (hybrid approach)

**Context:** LedgerPulse needs to extract structured data (line items, totals, tax amounts, dates) from invoice PDFs and photos. Options: (A) Local Tesseract.js (free, offline, lower accuracy), (B) Cloud API like Google Document AI (pay-per-page, high accuracy, requires internet), (C) Hybrid.

**Decision:** Use **Tesseract.js** as the default OCR engine (free, no API keys, works offline) for Phase 1–2. Provide an optional **Google Document AI / Azure Form Recognizer** integration as a premium feature toggle for users who need higher accuracy.

For extraction (post-OCR), use **regex + heuristics** for structured fields (invoice number, totals, dates), and reserve LLM-based extraction (e.g., GPT-4o, local Ollama) as a premium Phase 4+ option.

**Trade-offs:**
| Pro | Con |
|-----|-----|
| Tesseract.js: $0 cost, no API dependency | Lower accuracy on low-quality photos |
| Tesseract.js: Works fully offline | No native table extraction (must post-process) |
| Cloud fallback: >95% accuracy on structured docs | Cost scales with volume ($0.015/page for Doc AI) |
| Hybrid: user chooses based on need | Integration complexity of switching engines |

**Implementation:**
```typescript
// src/services/ocr/engine.ts
interface OcrEngine {
  extract(filePath: string): Promise<OcrResult>;
}

class TesseractEngine implements OcrEngine { ... }
class GoogleDocAiEngine implements OcrEngine { ... }

// Selected by config:
const engine: OcrEngine = config.ocrEngine === 'google'
  ? new GoogleDocAiEngine()
  : new TesseractEngine();
```

### ADR-004: Express vs Fastify

**Status:** Accepted

**Context:** The backend needs a Node.js HTTP framework. Express is the market default with the largest ecosystem. Fastify offers better performance (2–3x req/s) and built-in schema validation with JSON Schema.

**Decision:** Use **Express** for Phase 0–3. The primary reason is ecosystem maturity and the vast availability of middleware (multer, helmet, express-rate-limit, etc.). The performance difference is negligible at the expected scale (hundreds, not millions, of requests per day).

**Trade-offs:**
| Pro | Con |
|-----|-----|
| Largest middleware ecosystem | Slower than Fastify (~2x) for high concurrency |
| Every developer knows Express | No built-in schema validation (must use Zod manually) |
| Excellent TypeScript support (DefinitelyTyped) | Request/response serialization is not optimized |
| `express-async-errors` for clean error handling | |

**Migration path to Fastify (if needed):**
- The modular service architecture means only the `routes/` layer changes
- Controllers return identical responses regardless of framework
- Fastify's `fastify-express` plugin allows progressive migration

### ADR-005: JWT Session Auth Approach

**Status:** Proposed (Phase 5)

**Context:** LedgerPulse needs authentication for multi-user access (admin, reviewer, auditor roles). Options: (A) Session-based (express-session + connect-redis), (B) JWT with short-lived access tokens + refresh tokens, (C) OAuth2 proxy (e.g., Ory Hydra).

**Decision:** Use **JWT with access + refresh tokens** for Phase 5. Access tokens are short-lived (15 min) and stateless. Refresh tokens are long-lived (7 days) and stored in Redis for revocability. This combines the stateless scalability of JWT with the security of server-side session invalidation.

**Trade-offs:**
| Pro | Con |
|-----|-----|
| Stateless access tokens — no DB lookup on each request | Token revocation requires a blacklist (Redis) |
| Works with mobile/SPA clients | Refresh token rotation adds complexity |
| Can embed role + tenant in payload | Payload size grows with claims |
| Standard `Authorization: Bearer` header | Must implement secure storage on client |

**Implementation sketch:**
```typescript
interface JwtPayload {
  sub: string;       // user id
  role: 'admin' | 'reviewer' | 'auditor';
  tenantId?: string; // multi-tenant in Phase 5+
  iat: number;
  exp: number;
}

// Access token: signed with HS256, 15 min expiry
// Refresh token: opaque string stored in Redis { key: refresh:{token}, value: userId }
// Rotation: each refresh invalidates old refresh token, issues new pair
```
